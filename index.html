שעה 1 - יסודות תקשורת רשתות ו-IP

⏱ 0:00 - 1:00
🎯 מטרת השעה
להבין מה זו רשת, מה זו כתובת IP, ולמה בכלל צריך DHCP ו-DNS. בלי הבסיס הזה - שום דבר אחר לא יהיה ברור.

1.1 מה זה Internet Protocol (IP)?

Internet Protocol זה אוסף כללים מוסכמים שמגדירים איך מידע נשלח ברשת. בלי כללים משותפים, מחשב אחד לא ידע להבין מחשב אחר - בדיוק כמו ששני אנשים שמדברים שפות שונות לא יכולים לתקשר.

כל מכשיר ברשת (מחשב, נייד, מדפסת, נתב, שרת) חייב כתובת IP - מספר ייחודי שמזהה אותו ברשת. בלי כתובת IP, אי אפשר לשלוח אליו או לקבל ממנו מידע.

IPv4 גרסה 4 של פרוטוקול IP - הפופולרית ביותר היום. כתובת בת 32 ביט בפורמט 192.168.1.10 (ארבעה מספרים 0-255 מופרדים בנקודות).
IPv6 גרסה 6 - חדשה יותר, 128 ביט, נראית כך: 2001:0db8:85a3::8a2e:0370:7334. נולדה כי IPv4 נגמר (כמעט).

1.2 כתובות פרטיות מול ציבוריות (Private vs Public)

🏠 Private IP (פרטית)

בשימוש ברשתות פנימיות שלא מחוברות ישירות לאינטרנט. כל ארגון יכול להשתמש בהן בלי לבקש רשות מאף אחד.

טווחים שמורים (RFC 1918):

  • 10.0.0.0 - 10.255.255.255
  • 172.16.0.0 - 172.31.255.255
  • 192.168.0.0 - 192.168.255.255

🌍 Public IP (ציבורית)

כתובות שמוקצות לכל מי שמתחבר לאינטרנט. כל כתובת ציבורית בעולם היא ייחודית.

כל ספק אינטרנט (ISP) מקצה ללקוחותיו כתובת ציבורית, ומספר ה-IPs הציבוריים בעולם מוגבל - לכן הומצא ה-NAT.

⚠️ למה זה חשוב לפנטסטרים?
בסקירה (recon) חשוב להבחין מיד אם רואים כתובת פרטית או ציבורית - זה אומר לך אם המכשיר חשוף ישירות לאינטרנט או יושב מאחורי NAT.

NAT - הגשר בין פנימי לחיצוני

Network Address Translation זה המנגנון שמאפשר למאות מכשירים עם כתובות פרטיות לצאת לאינטרנט דרך כתובת ציבורית אחת. הנתב הביתי שלך עושה NAT כל יום - הוא מתרגם את 192.168.1.X שלך לכתובת הציבורית של בזק/הוט/סלקום.

תרגום כתובות (NAT) - מספר מכשירים פנימיים יוצאים החוצה דרך כתובת ציבורית אחת

1.3 מחלקות IPv4 (Classful)

בעבר חילקו את טווח IPv4 למחלקות לפי גודל. כיום משתמשים בעיקר ב-CIDR (notation של /24 וכו'), אבל המונחים נשארו.

מחלקהPrefixטווח פרטימס' Hosts אפשרישימוש אופייני
Class A/810.0.0.0 - 10.255.255.25516,777,214ארגונים ענקיים, ספקי ISP
Class B/16172.16.0.0 - 172.31.255.25565,534ארגונים בינוניים-גדולים
Class C/24192.168.0.0 - 192.168.255.255254רשתות ביתיות, משרדים קטנים
💡 למה "מינוס 2"?
בכל סאבנט שתי כתובות שמורות:
  • כתובת הרשת (הראשונה, למשל 192.168.1.0) - מזהה את הרשת עצמה.
  • כתובת ה-Broadcast (האחרונה, למשל 192.168.1.255) - שידור לכל הרשת.
לכן ב-/24 יש 256 כתובות אבל רק 254 שמישות ל-hosts.

1.4 כתובות מערכת מיוחדות

🔄 Loopback - 127.0.0.0/8

כתובת שמייצגת את המחשב עצמו. כל מה שנשלח אליה חוזר אליה. הכתובת המוכרת ביותר היא 127.0.0.1.

שימושים נפוצים:

  • בדיקה שהמחסנית הרשתית של המחשב עובדת.
  • הרצת שרת מקומי שרק המחשב עצמו ניגש אליו (למשל פיתוח אתר ב-http://127.0.0.1:8080).
  • גם השם localhost מצביע עליה.

🚫 APIPA - 169.254.0.0/16

Automatic Private IP Addressing - מה שקורה כאשר המחשב לא מצליח להגיע ל-DHCP. הוא מקצה לעצמו כתובת מהטווח 169.254.X.X כדי שיוכל לכל הפחות לתקשר עם מכשירים אחרים באותה רשת מקומית.

🚨 איך מזהים בעיית DHCP?
אם פקודת ipconfig מראה לך כתובת שמתחילה ב-169.254. - אין תקשורת עם DHCP. סיבות אפשריות: כבל לא מחובר, DHCP נפל, הרשת מנותקת מהשרת, הגדרת VLAN שגויה.
📝 בדיקה עצמית:
איזו כתובת היא פרטית? 10.5.20.1 או 74.125.224.72?

10.5.20.1 - היא בתוך טווח Class A הפרטי (10.0.0.0/8). השנייה היא כתובת ציבורית של גוגל.

מה זה אומר שמחשב קיבל את הכתובת 169.254.10.5?

שהמחשב לא הצליח להגיע לשרת DHCP, אז הוא הקצה לעצמו כתובת APIPA. צריך לבדוק את החיבור והגדרות הרשת.

שעה 2 - הכרת DHCP ותהליך DORA

⏱ 1:00 - 2:00
🎯 מטרת השעה
להבין למה צריך DHCP, איך הוא עובד בפועל, ומה זה תהליך DORA - הליבה של איך המחשב שלך מקבל IP אוטומטית.

2.1 מה זה DHCP?

DHCP - Dynamic Host Configuration Protocol זה פרוטוקול ניהול רשת שמחלק אוטומטית הגדרות רשת (IP, Subnet Mask, Default Gateway, DNS) למכשירים בכל פעם שהם מתחברים לרשת.

פורט DHCP השרת מאזין על UDP port 67, הלקוח על UDP port 68. שניהם UDP, לא TCP - כי הפרוטוקול עובד ב-broadcast לפני שיש בכלל IP.

למה זה כל-כך חשוב?

תארו לעצמכם ארגון עם 500 מחשבים. בלי DHCP:

  • חייב אדמין לקבוע ידנית כתובת IP, Subnet, Gateway, DNS לכל אחד.
  • טעויות בקלדה גורמות להתנגשויות IP (שני מחשבים עם אותה כתובת).
  • כל שינוי ברשת (החלפת DNS למשל) - עוברים מכשיר-מכשיר.
  • אורח שמגיע עם נייד? צריך אדמין שיגדיר לו ידנית.

עם DHCP - הכל אוטומטי, מהיר ובלי טעויות.

2.2 הגדרה דינמית מול סטטית

⚡ Dynamic (DHCP)

  • הגדרה אוטומטית בעלייה.
  • אחוז גבוה מאוד של תצורה נכונה.
  • עדכון אוטומטי לכל הלקוחות בשינוי.
  • מבטל מקור נפוץ לתקלות רשת.

📌 Static (ידני)

  • תצורה מוגדרת ידנית.
  • סיכון לטעויות אנוש.
  • שינויים גורמים לתקלות תקשורת.
  • מאמץ אדמיניסטרטיבי גבוה.
📌 מתי בכל זאת משתמשים בסטטי?
  • שרתים חשובים (DC, DNS, קבצים, מדפסות) - שלא ישתנו IP.
  • נתבים ו-Switches.
  • מכשירי IoT/בקרת תעשייה (מצלמות, בקרים).
  • שרת ה-DHCP עצמו!
פתרון ביניים: DHCP Reservation - שומרים לכתובת MAC ספציפית את אותה כתובת IP תמיד. נלמד את זה בשעה הבאה.

2.3 תהליך DORA - הלב של DHCP

DORA = Discover, Offer, Request, Acknowledge - ארבעת השלבים שלקוח עובר כדי לקבל כתובת IP.

תהליך DORA - 4 הודעות בין הלקוח לשרת
  1. D - DHCPDISCOVER (גילוי): הלקוח עולה לראשונה לרשת בלי IP. הוא שולח חבילת broadcast (לכל הרשת, כתובת יעד 255.255.255.255) שאומרת בעצם "מישהו פה DHCP? אני צריך IP". כתובת המקור היא 0.0.0.0 כי אין לו עדיין כתובת.
  2. O - DHCPOFFER (הצעה): כל שרת DHCP שמקבל את ה-DISCOVER עונה עם הצעה - "אני יכול להציע לך את הכתובת X לתקופה Y". ההצעה מגיעה גם ב-broadcast (כי ללקוח עדיין אין כתובת רגילה).
  3. R - DHCPREQUEST (בקשה): הלקוח בוחר את ההצעה הראשונה שהגיעה (אם יש כמה שרתים). הוא שולח שוב broadcast עם בקשה רשמית עבור הכתובת. למה broadcast? כדי שכל שאר השרתים שהציעו יידעו שהוא בחר אחר ויחזירו את הכתובות לבריכה.
  4. A - DHCPACK (אישור): השרת רושם בבסיס הנתונים שלו שהכתובת תפוסה, ושולח אישור סופי עם כל הפרטים (IP, Subnet Mask, Gateway, DNS, lease time).
  5. (שלב 5 - אימות): הלקוח שולח בקשת ARP כדי לוודא שאף אחד אחר ברשת לא משתמש כבר באותה כתובת. אם אין תגובה - הכל תקין.
🔐 נקודה חשובה לאבטחה
כל ההודעות הראשונות הן broadcast לא מאומת. תוקף ברשת יכול להציע IP לפני השרת האמיתי - התקפת DHCP Spoofing. מגנים על זה ב-Switches עם DHCP Snooping.

2.4 DHCP Lease - השכרת הכתובת

כתובת IP לא ניתנת לתמיד - היא מושכרת לזמן מוגדר. ברירת המחדל ב-Windows DHCP היא 8 ימים. בסיום ה-lease, הכתובת חוזרת לבריכה ויכולה להינתן ללקוח אחר.

תהליך חידוש (Renewal)

  1. 50% מה-lease: הלקוח מנסה לחדש מול אותו שרת ששכר ממנו. אם השרת עונה - מקבל lease חדש לזמן מלא.
  2. 75% מה-lease (T2): אם לא קיבל תשובה, הלקוח מנסה broadcast לכל שרת DHCP ברשת.
  3. 100% מה-lease: אם עדיין אין תשובה - הכתובת מתבטלת. הלקוח מתחיל מההתחלה (DORA חדש), ואם גם זה נכשל - מקבל APIPA.
⚙️ איך לבחור lease time?
  • רשת יציבה (משרדים, שולחנות עבודה): lease ארוך - 8 ימים ומעלה. פחות תעבורה, פחות עומס.
  • רשת דינמית (מלון, אורחים, Wi-Fi פתוח): lease קצר - שעה עד יום. כתובות חוזרות מהר לבריכה.
  • כלל אצבע: ככל שיותר מכשירים נכנסים ויוצאים - lease קצר יותר.

שעה 3 - התקנה והגדרת DHCP Server על Windows Server

⏱ 2:00 - 3:00
🎯 מטרת השעה
להתקין שרת DHCP מ-A ל-Z על Windows Server, להגדיר Scope, Exclusions, Reservations ו-Filters. בסוף השעה - כל תלמיד יידע להקים שרת DHCP פונקציונלי.
⚙️ דרישות מקדימות
  • Windows Server (2019 / 2022 / 2025) מותקן ופועל.
  • כתובת IP סטטית לשרת עצמו (שרת DHCP חייב IP קבוע!).
  • חיבור לדומיין (אם רוצים אוטוריזציה דרך AD - הכי מקובל).
  • הרשאות Domain Admin / Enterprise Admin.

3.1 התקנת התפקיד (Role)

  1. פתיחת Server Manager: לחיצה על תפריט Start ⟵ Server Manager. במסך הראשי לוחצים על Manage בפינה הימנית העליונה ⟵ Add Roles and Features.
  2. Before You Begin: מסך פתיחה. לוחצים Next. (אפשר לסמן Skip this page by default כדי לדלג בעתיד.)
  3. Installation Type: בוחרים Role-based or feature-based installation ⟵ Next.
  4. Server Selection: בוחרים את השרת מהרשימה (בדרך כלל יש רק אחד) ⟵ Next.
  5. Server Roles: ✅ מסמנים את התיבה ליד DHCP Server. יקפוץ חלון שמבקש להוסיף features תומכות - לוחצים Add Features ⟵ Next.
  6. Features: משאירים כברירת מחדל ⟵ Next.
  7. DHCP Server: מסך מידע. Next.
  8. Confirmation: מסמנים Restart the destination server automatically if required (אופציונלי) ⟵ לוחצים Install. ההתקנה לוקחת 2-5 דקות.

3.2 הרשאה מול ה-DC (Authorization)

בסביבת דומיין, שרת DHCP חייב להיות מאושר על ידי הדומיין כדי להקצות כתובות. זה הגנה מובנית של Microsoft מפני שרתי DHCP זדוניים (Rogue DHCP).

  1. אחרי ההתקנה - בדגל הצהוב למעלה ב-Server Manager תופיע הודעה Complete DHCP Configuration. לחיצה עליה.
  2. Description: מסך מידע ⟵ Next.
  3. Authorization: בוחרים את המשתמש שיבצע את האוטוריזציה (ברירת מחדל - המשתמש המחובר, צריך להיות Enterprise Admin) ⟵ Commit.
  4. Summary: אמורים לראות שני סטטוסים של "Done":
    • Creating security groups: Done
    • Authorizing DHCP server: Done
    לוחצים Close.
🚨 שגיאה נפוצה
אם השרת לא מאושר - הוא יותקן, אך לא ישיב לבקשות DHCP. בדיקה: ב-DHCP Manager, אם רואים חץ אדום על השרת ⟵ הוא לא מאושר. פתרון: לחיצה ימנית על השרת ⟵ Authorize.

3.3 יצירת Scope - בריכת הכתובות

Scope זה הגדרה של טווח כתובות שהשרת יכול להקצות. בלי Scope - השרת קיים אבל לא מחלק כלום.

  1. פתיחת DHCP Manager: ב-Server Manager ⟵ Tools ⟵ DHCP.
  2. ניווט ל-IPv4: בחלונית הימנית, פותחים את העץ: שם השרת ⟵ IPv4.
  3. יצירת Scope חדש: לחיצה ימנית על IPv4 ⟵ New Scope....
  4. Name: שם תיאורי, למשל Office_LAN_192.168.1.0. תיאור אופציונלי. ⟵ Next.
  5. IP Address Range: ממלאים את הטווח:
    • Start IP address: 192.168.1.10
    • End IP address: 192.168.1.200
    • Length: 24
    • Subnet mask: ימולא אוטומטית - 255.255.255.0
    ⟵ Next.
  6. Add Exclusions and Delay: כתובות בתוך הטווח שלא יוקצו (למשל מדפסות סטטיות). אפשר לדלג ולחזור אחר כך. ⟵ Next.
  7. Lease Duration: ברירת המחדל 8 ימים. אפשר לשנות לפי הצורך. ⟵ Next.
  8. Configure DHCP Options: בוחרים Yes, I want to configure these options now ⟵ Next.
  9. Router (Default Gateway): ממלאים את כתובת הנתב (192.168.1.1) ⟵ Add ⟵ Next.
  10. Domain Name and DNS Servers: Parent Domain: cyber.local. בשורת ה-IP address של DNS - מוסיפים את כתובת ה-DNS של הדומיין (לרוב ה-DC עצמו). Add ⟵ Next.
  11. WINS Servers: בדרך כלל מדלגים (WINS פרוטוקול ישן) ⟵ Next.
  12. Activate Scope: בוחרים Yes, I want to activate this scope now ⟵ Next ⟵ Finish.

3.4 Exclusions - הוצאת כתובות מהבריכה

אם יש לך כתובות בתוך הטווח שכבר תפוסות סטטית (למשל מדפסות, מתגים), צריך להוציא אותן מהבריכה כדי שה-DHCP לא יקצה אותן בטעות.

  1. ב-DHCP Manager ⟵ פותחים את ה-Scope ⟵ לחיצה ימנית על Address Pool ⟵ New Exclusion Range.
  2. ממלאים Start IP ו-End IP (לכתובת בודדה ממלאים רק Start) ⟵ Add ⟵ Close.

3.5 Reservations - הקצאה קבועה ב-MAC

Reservation נותנת לכתובת MAC ספציפית תמיד את אותה כתובת IP, גם שהיא דרך DHCP. שילוב מנצח: יציבות של סטטי + ניהול מרכזי של DHCP.

📌 דוגמאות לשימוש ב-Reservation
  • מדפסת רשת - שתמיד באותה כתובת לקיצורי דרך.
  • שרת קבצים פנימי.
  • מצלמות אבטחה - שכתובות יהיו צפויות.
  • שולחן עבודה של מנהל / מפתח שדורש פורט פתוח.
  1. בתוך ה-Scope ⟵ לחיצה ימנית על Reservations ⟵ New Reservation.
  2. ממלאים:
    • Reservation name: שם תיאורי, למשל Printer-HR.
    • IP address: הכתובת הקבועה (חייבת להיות בתוך ה-Scope).
    • MAC address: כתובת ה-MAC של המכשיר (אפשר ללא מקפים, למשל AABBCC112233).
    • Description: תיאור אופציונלי.
    • Supported types: Both (DHCP + BOOTP) - בטוח לבחור.
  3. Add ⟵ Close.

טיפ: כדי למצוא MAC של מכשיר Windows - ipconfig /all, השדה Physical Address.

3.6 DHCP Filters - שליטה גרעינית במי מקבל

Filter מגביל את ההקצאה לפי MAC. שני סוגים:

  • Allow filter (Whitelist): כשמופעל - רק מכשירים ברשימה מקבלים IP. הכי מחמיר.
  • Deny filter (Blacklist): כשמופעל - מכשירים ברשימה לא מקבלים. כל השאר מקבלים.
  1. ב-DHCP Manager ⟵ פותחים IPv4 ⟵ Filters ⟵ Allow או Deny.
  2. לחיצה ימנית ⟵ New Filter ⟵ ממלאים MAC + תיאור ⟵ Add.
  3. כדי להפעיל: לחיצה ימנית על Allow או Deny ⟵ Enable.
⚠️ זהירות עם Allow Filter
ברגע שמפעילים Allow ולא הזנת אף MAC - אף אחד לא יקבל IP יותר! תמיד למלא MACs קודם, להפעיל אחר כך.

שעה 4 - הכרת DNS, אזורים ו-Records

⏱ 3:00 - 4:00
🎯 מטרת השעה
להבין מה זה DNS, איך הוא ממיר שמות לכתובות, מה זו ההיררכיה של הדומיינים, מה זה Forward/Reverse Lookup ואיזה סוגי Records קיימים.

4.1 מה זה DNS?

DNS - Domain Name System הוא פרוטוקול שממיר שמות דומיין (כמו www.google.com) לכתובות IP (כמו 142.250.190.78). זה ה"ספר טלפונים" של האינטרנט.

פורט DNS UDP port 53 (לשאילתות רגילות), TCP port 53 (לתשובות גדולות, zone transfer, DNS-over-TLS).
📌 למה DNS קריטי בארגון?
ב-Active Directory, בלי DNS פעיל - אין דומיין. הלקוחות מאתרים את ה-DC דרך DNS records ספציפיים (SRV records). מחשב שלא מגיע ל-DNS - לא יכול להתחבר לדומיין, לא יכול לעדכן GPO, לא יכול לבצע אימות Kerberos.

4.2 יתרונות וחסרונות

✅ יתרונות

  • גלישה נוחה - לא צריך לזכור IP.
  • שינוי IP של שרת בלי לפגוע במשתמשים.
  • Load Balancing - אותו שם, כמה כתובות.
  • DNS מוצפן (DoH/DoT) - שיפור פרטיות.
  • מסייע במניעת התקפות בהגדרה נכונה.

⚠️ חסרונות / סיכונים

  • נקודת כשל - DNS למטה = רשת למטה.
  • פגיע ל-DDoS.
  • חשוף ל-DNS Poisoning / Spoofing.
  • שאילתות לא מוצפנות נחשפות בדרך.
  • ICANN שולטת בקצוות העליון של המערכת.

4.3 מבנה FQDN - שם דומיין מלא

FQDN = Fully Qualified Domain Name. שם מלא, מקצה לקצה.

חלקי ה-FQDN - נקראים מימין לשמאל בהיררכיית DNS

שלוש רמות הדומיין

Root Domain השורש - הנקודה הבודדה בסוף. יש בעולם 13 שרתי root (A עד M) שמובילים את כל ה-DNS. בכל שאילתה מלאה הנקודה מובלעת בסוף.
TLD - Top Level Domain הסיומת: .com, .org, .gov, .il, .co.il. שני סוגים: gTLD (גנרי) ו-ccTLD (לפי מדינה).
SLD - Second Level החלק שהארגון רוכש: hackeru, google, microsoft. תחתיו אפשר ליצור עוד תת-דומיינים: www, mail, shop.

4.4 איך נראית שאילתת DNS אמיתית?

כאשר אתה מקליד www.hackeru.com בדפדפן, מאחורי הקלעים קורה תהליך מסובך:

  1. בדיקת cache מקומי: המחשב בודק קודם את ה-cache שלו (ipconfig /displaydns) ואת קובץ ה-hosts. אם יש - חוזר מיד.
  2. שאילתה ל-DNS המקומי: אם אין ב-cache, נשלחת שאילתה ל-DNS המקומי (בדרך כלל הראוטר או DC). זה ידוע כ-Recursive Query - "בבקשה מצא לי את התשובה".
  3. שאילתה ל-Root: אם ה-DNS המקומי לא מכיר - הוא פונה לאחד מ-13 שרתי הroot ושואל "מי אחראי על .com?". מקבל הפניה לשרתי TLD.
  4. שאילתה ל-TLD: ה-DNS המקומי שואל את שרת .com "מי אחראי על hackeru.com?". מקבל הפניה ל-DNS האוטוריטטיבי של hackeru.
  5. שאילתה ל-Authoritative: שאלה אחרונה - "מה הכתובת של www.hackeru.com?". התשובה מגיעה: 199.244.49.102.
  6. חזרה ללקוח + caching: ה-DNS המקומי שומר את התשובה ב-cache (לפי TTL) ומחזיר ללקוח. בפעם הבאה - יענה מיד מה-cache.
תהליך resolution היררכי - מהשורש עד התשובה

4.5 Forward Lookup ו-Reverse Lookup

➡️ Forward Lookup

שם ⟵ IP

"מה הכתובת של www.hackeru.com?" ⟵ 199.244.49.102.

זה השימוש הנפוץ - 99% מהשאילתות. גלישה, שליחת מייל, חיבור לשרת.

⬅️ Reverse Lookup

IP ⟵ שם

"איזה שם משויך ל-10.10.10.10?" ⟵ DC-01.cy.local.

שימושי ב-logs, ניטור, חוקרי אבטחה, מערכות אנטי-spam (בודקות שמייל מגיע משרת תקין).

4.6 סוגי Zones

Primary Zone

אזור ראשי - העותק המקורי הניתן לכתיבה. כל השינויים מתבצעים פה. ניתן לעדכן אוטומטית או ידנית.

Secondary Zone

אזור משני - עותק לקריאה בלבד. מעודכן דרך Zone Transfer מה-Primary. מטרה: חיזוק עמידות וביצועים.

Stub Zone

אזור מקוצר - מכיל רק את ה-NS records לזיהוי איזה DNS אחראי על האזור. שימוש: הפניות מהירות בלי לשמור את כל הנתונים.

4.7 DNS Records - סוגי הרישומים

SOA

Start of Authority - הרשום הראשון בכל zone. מי האוטוריטטי, מספר סידורי, זמני רענון, TTL מוגדר.

A

Address - מיפוי שם ⟵ IPv4. server.cyber.local ⟵ 10.0.0.5. הנפוץ ביותר.

AAAA

נקרא "Quad-A". מיפוי שם ⟵ IPv6. אותו דבר כמו A, רק לכתובות 128-bit.

CNAME

Canonical Name - כינוי. mail.cyber.local ⟵ webmail.cyber.local. מצביע על שם אחר, לא על IP.

MX

Mail Exchanger - שרת המייל של הדומיין. כשמישהו שולח לך מייל - השולח שואל את DNS שלך "MX?".

NS

Name Server - איזה DNS אוטוריטטיבי על הדומיין. בכל zone יש לפחות אחד.

PTR

Pointer - מיפוי IP ⟵ שם (היפוך של A/AAAA). נמצא ב-Reverse Lookup Zone.

SRV

Service - מאתר שירותים בדומיין. ה-DC משתמש בזה: _ldap._tcp.cyber.local מצביע על שרתי הדומיין.

TXT

טקסט חופשי. בשימוש למשל ב-SPF, DKIM, אימותי בעלות, וגם רישומי אבטחה.

4.8 Forwarders - תקשורת ל-DNS חיצוני

בארגון יש כמה שרתי DNS. במקום שכל אחד יבצע resolution לעולם החיצון - מגדירים Forwarder.

Forwarder שרת DNS שמטפל בכל הבקשות שיוצאות מהארגון. אם DNS פנימי לא יודע את התשובה - הוא שולח ל-forwarder. רק ה-forwarder ניגש לעולם. שיפור אבטחה ויעילות.
Conditional Forwarder הפניה תלוית-תנאי. למשל: כל שאילתה על partner.com תופנה ל-DNS של השותף, ולא דרך ה-forwarder הרגיל. שימושי בחיבורי VPN בין ארגונים.

5.5 יצירת Reverse Lookup Zone

⚠️ לא נוצר אוטומטית!

  1. ב-DNS Manager ⟵ לחיצה ימנית על Reverse Lookup Zones ⟵ New Zone....
  2. סוג: Primary zone ⟵ Next.
  3. IPv4 או IPv6 ⟵ בוחרים IPv4 ⟵ Next.
  4. Network ID: מקלידים את שלושת המספרים הראשונים של הרשת. למשל לרשת 10.0.0.0/24 ⟵ 10.0.0. שם האזור יווצר אוטומטית: 0.0.10.in-addr.arpa.
  5. בחירת מצב Dynamic Update ⟵ Next ⟵ Finish.
📌 למה השם הפוך?

הוספת PTR Record ידני

  1. פותחים את ה-Reverse Zone ⟵ לחיצה ימנית ⟵ New Pointer (PTR)....
  2. Host IP Address: 10.0.0.5. Host name: fileserver.cyber.local. ⟵ OK.

  3. יצירת Reverse Lookup Zone (חובה ידנית!).

🚨 פינת אבטחה - מבט פנטסטר

  • DHCP Spoofing/Starvation: תוקף ברשת מנצל את ההצעות הראשונות. הגנה: DHCP Snooping בסוויצ'.
  • Rogue DHCP: שרת DHCP זדוני מציע IP + DNS זדוני. הגנה: אוטוריזציה ב-AD, רק שרת מאושר עונה.
  • DNS Cache Poisoning: הזרקת רשומה שקרית ל-cache. הגנה: DNSSEC, DoH/DoT.
  • DNS Tunneling: נתונים מוסתרים בתוך שאילתות DNS לעקיפת חומת אש. ניטור על אורך וכמות.
  • Zone Transfer לא מאובטח: תוקף מבקש AXFR ומקבל את כל ה-records. בדיקה: dig axfr @ns1 domain.com.
